# COPTA — PRD（開発用）

> 戦略の詳細は `docs/STRATEGY.md`（Claude案 + Gemini良案 + GPT良案を統合）を参照。
> このファイルはCursorでの実装着手用に、MVPスコープだけを実行可能な粒度に落としたもの。

## プロダクト一言定義
PTAが紙・現金・LINE・Excelでやっていた運営業務を、ITが苦手な保護者でも回せる「運営OS」に置き換える。
連絡網では戦わない。差別化の核は **集金（法令適合の収納代行）＋ 役職に紐づく引き継ぎ・会計**。

## MVPスコープ（Phase 0：最初の6ヶ月）

実装順は「連絡先行・集金は設計を先に固める」。PMF確認後すぐ集金を載せられるよう、データモデルは最初から集金前提で作る。

### 含む機能
1. 団体・名簿セットアップ（学校／クラス／委員会の階層）
2. 子どもID方式の自己登録（保護者は学校発行コードで自己登録。役員は「登録済みか」のみ閲覧、詳細はマスク＝ゼロトラスト名簿）
3. 役職ベースの権限管理（個人ではなく役職に権限。4/1で自動ダウングレード）
4. お知らせ配信・回覧板（既読確認、要返信フラグ、セグメント配信、承認ワークフロー、**コメント機能は実装しない**＝荒れる温床になるため）
5. 出欠確認・アンケート（テンプレ、自動集計）
6. 引き継ぎボックス（年度をまたいで残る。無料プランでも提供＝口コミの起点）
7. 役員間チャット（役職・委員会単位のみ。全保護者自由投稿は実装しない）
8. 未登録者抽出 → 紙配布用レイアウト出力（非スマホ層対応）
9. ボランティア募集（人数・日時・先着/抽選・キャンセル待ち）

### 明示的に含まない（Phase 0では作らない）
- 決済実行（UIの請求作成までは作るが、実際の入金フローはPhase間近で接続）
- 地域広告・クーポン
- 当番自動編成、バザー物販決済
- 子ども本人アカウント、学校成績・出席情報の取り込み
- 自由なタイムライン／全体DM
- 自前の前払式支払手段（チャージ残高）

## 認証・登録フロー
- パスワードレス（マジックリンク or SMS使い捨てコード）
- 学校/PTAが「子どもID（QRコード/コード文字列）」を子ども単位で発行 → 保護者がそれを読み取って自己登録
  （※クラス共通の招待コードだけにすると「誰の保護者か」の紐付けが弱くなり、会費請求の正確性が落ちる。
  必ず子ども個別のコードにする）
- **登録は5分以内で完了**することをUX目標にする（子ども追加 → 学校選択 → 学年・組 → 子どもIDコード入力 → 完了）
- 1アカウントで複数の子ども（兄弟）を紐付け可能
- 4/1ロールオーバー：前年度の役職保持者は自動的に一般保護者権限へダウングレード。新役職者に権限が引き継がれる（データは役職に紐づくため移行不要）

## 役員ダッシュボード（最重要画面）
保護者向けより、ここに開発の重心を置く。役員が毎月見るのはこの4つの数字だけで十分。
- 未払い人数
- 未読人数
- 今週のイベント件数
- アンケート回答率
複雑な分析画面は作らない。この4指標がワンビューで見えることを優先する。

## Phase 3候補（MVPには含まないが設計上ぶつからないようにする）
- **防災モード**：災害時のみ起動する特別表示（避難所情報・安否確認・給水/炊き出し情報）。自治体の関心を引く独立した訴求軸になるが、「公式防災インフラ」としての可用性責任を負うと重くなるため、自治体公式系の補助・PTA活用の位置づけに留める。MVPの認証・通知設計がこの拡張を阻害しないようにしておく程度で良い。

## 権限モデル（重要：個人ではなく役職に紐づける）
```
User (保護者) ─ has_many → OrganizationMembership ─ belongs_to → Organization (PTA団体)
  └─ Organization ごとに:
       └─ Role (会長 / 副会長 / 会計 / 広報委員 / 学級委員 / 監査 / 一般保護者)
            └─ Permission set（閲覧範囲・操作範囲）
       └─ User ─ has_many → RoleAssignment (school_year付き、Organization内のRoleを参照)
       └─ Child (子ども) ─ belongs_to → User(s), Class ─ status: active/graduated/withdrawn
```
- 1つの`User`が複数の`Organization`（例：小学校PTA＋中学校PTA）に同時所属できる（`organization_memberships`、§ライフサイクル設計参照）。
- RoleAssignmentは `school_year` を持つ。年度が変わると新しいRoleAssignmentが作られ、旧年度のものは失効する（履歴は保持＝引き継ぎ・監査ログに使う）。
- データ（お知らせ下書き、引き継ぎメモ、会計帳簿）は `Role` または `Organization` に紐づき、`User`には直接紐づけない。これが「人が辞めても消えない」設計の根幹。

## 画面構成（MVP）
- 保護者用：ホーム（要対応バッジ）／お知らせ／出欠・アンケート／マイ所属
- 役員用：上記＋配信作成／名簿管理／引き継ぎ／権限管理（役職切替で見える範囲が変わる）

## ライフサイクル設計：卒業・進学・きょうだい（重要）

このプロダクトのユーザーは、PTAの一般SaaSと違って**100%が「いずれ卒業する」前提のユーザー**である。これを離脱（チャーン）として扱わず、「進学先でも継続利用される」成長導線として設計する。

### 基本モデル
- **User（保護者アカウント）は特定の学校・PTAに固定しない。** `organization_memberships` で複数組織に同時所属できる。
  - きょうだいが別の学校（例：小学校＋中学校）に通う場合、1つの保護者アカウントで両方のPTAに同時所属できる。
  - 上の子が卒業して中学に上がり、その中学も本プラットフォームを使っていれば、**既存アカウントに新しい組織membershipを追加するだけ**で継続利用できる（アカウント新規作成不要）。これは離脱阻止であると同時に、学校間の自然な拡大導線になる。
- **Child（子ども）は卒業しても削除しない。** `children.status` を `active → graduated` に変更し、`graduated_at` を記録する。
  - 卒業した子ども宛の新規お知らせ・新規集金は発生しなくなる。
  - 過去の支払い履歴・領収書・引き継ぎ記録は**読み取り専用で残す**（税務・記録目的、保護者からの要望が多い）。
- **組織側の卒業判定**：`organizations.final_grade_label`（例：小学校なら"6年"）と `children.grade_label` を突き合わせ、年度末に**バッチで自動的に卒業ステータスへ遷移**させる。学校種別ごとに最終学年が違う（幼保・小・中・高）ため、組織単位で設定可能にする。
- **保護者の組織membership**：その組織に紐づく子どもが全員卒業/転校したら `organization_memberships.status = 'inactive'`（退会ではなくアーカイブ）。再度同じ学校に下の子が入学すれば `active` に戻せる。
- **役職（Role）との関係**：卒業＝即座に役職を失うわけではない（年度途中で子が卒業するケースは稀だが、年度末の通常ロールオーバーで自然に役職も終了する）。既存の `role_assignments` の年度スコープ機構がそのまま使える。

### きょうだいがいる場合のUX
- ホーム画面は**子どもごとに分けず、1つのフィードに統合**し、子ども名のタグで区別する（「今日の提出物：太郎（3組）／花子（1組）」のように）。
- 通知も子どもごとに別送せず、**1日1ダイジェストに束ねる**（GPTの通知設計指摘を踏襲。子が3人いるからといって通知が3倍になるのは離脱要因になる）。
- 集金は子どもごとの請求項目を出しつつ、**合算して1回の決済で支払える**ようにする。兄弟割（減免）はOrganization単位の設定で対応（§7-3で既出）。
- 子どもが別々の学校（＝別Organization）に通う場合でも、ホーム画面は組織をまたいで統合表示する（保護者から見て「学校が違う」ことを意識させない）。

### 退会・データ削除
- 全ての子どもが卒業し、保護者が明示的に退会を申請した場合のみ、個人情報を削除する。デフォルトでは自動削除しない（領収書参照などの実需があるため、また次の子の入学時に再アクティブ化できる利便性を優先）。
- 削除リクエストフローはPhase 1で実装（MVPでは「问い合わせで対応」運用でも可）。

### Phase 2/3候補：卒業後の地域住民モード
- 全員卒業した家庭でも、地域に住み続ける限り「地域モード」（チラシ・クーポン・地域イベント情報）には価値が残る可能性がある。`organization_memberships.status = 'inactive'` のユーザーに対し、PTA機能は閉じつつ地域コミュニティ機能だけ持たせる「卒業生・地域住民」アカウント種別は将来検討（MVPでは作らない。まずはPTA運営OSとしての価値を固めることを優先）。


- パイロット5〜10校で保護者登録率 80%以上
- 翌年度継続率 100%（役員交代後も解約されない）
- 学校からの反対ゼロ（学校無関与設計が機能しているか）
